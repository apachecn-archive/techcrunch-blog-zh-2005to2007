# 新网络计算&面向服务的客户端

> 原文：<https://web.archive.org/web/http://techcrunch.com/2006/11/09/neo-network-computing-the-service-oriented-client/>

谷歌的 Eric Schmidt 昨天在 O'Reilly 的 web 2.0 会议上为 webware 和基于 Web 的应用做了一个完美的类比:

> 从根本上说，把钱存在银行里比放在口袋里更好

Schmidt 是在评论这样一种观点，即最近的建筑发展已经使 Ellison 的[网络计算机](https://web.archive.org/web/20160420163851/http://en.wikipedia.org/wiki/Network_Computer_Reference_Profile)的愿景最终得以实现；他在 Sun 和 Google 的贡献当然值得一提。然而，银行**非常**可用——我们不必担心消失和重新出现的道路或自动取款机！

尽管我们已经看到了基于 web 的应用程序的创新性飞跃，但是客户端假定与服务器的连接是持久的，并且很少以一致的方式适应偶尔的或丢失的连接。尽管 AJAX 的兴起，web 应用程序仍然主要是围绕请求页面而不是请求远程服务来构建的。今年早些时候，我听到 Adobe 的 Kevin Lynch 谈到对面向服务的客户端的需求，这种客户端可以更加智能地利用本地客户端服务，并包括对远程服务的一致支持。这有助于带来更丰富的用户体验，以及可以继续运行的应用程序，或者在连接不稳定或不存在时优雅地降级。

具有讽刺意味的是，微软或许正在为更丰富的面向服务的客户端引领潮流，例如 [*Live Writer*](https://web.archive.org/web/20160420163851/http://windowslivewriter.spaces.live.com/) 和[*Live Mail Desktop*](https://web.archive.org/web/20160420163851/http://ideas.live.com/programpage.aspx?versionId=6ac2bed2-b5a4-4a0a-a897-e36dd191a9f4)。我希望看到谷歌更充分地拥抱这种范式，但这里有一个潜在的第二产业，软件开发人员可以利用开放 API，为领先的应用程序创建面向服务的客户端，如[](https://web.archive.org/web/20160420163851/http://gmail.google.com/)**等。**

 **就我个人而言，我希望有一个[*Dreamweaver*](https://web.archive.org/web/20160420163851/http://www.adobe.com/products/dreamweaver/)+[*Typepad*](https://web.archive.org/web/20160420163851/http://www.typepad.com/)面向服务的客户端来管理我的博客:)**
