**5。[克里斯被写到近因](https://web.archive.org/web/20220817045306/http://www.chriswere.com/?p=42)**

“最近是一种新兴技术，对网络来说是(相对)新的东西。它的主要驱动力是 RSS，它提供了一个非常简单的机制来告诉世界什么时候是新的或者已经改变了。最初主要是作为博客交流新故事的工具，现在正迅速发展到涵盖新的领域；主流媒体、音频(播客)、视频等。有很多理由相信这只是一个开始——特别是考虑到微软宣布 RSS 也将应用于“列表”。在他们的新操作系统中，将 RSS 带入主流。

那么，新近有什么特别之处呢？目前，任何人都可以在谷歌上搜索任何东西，它会显示相关结果的列表——但现在呢？我想找到过去 24/12/2 小时内的结果。我们所知的网络始于静态网页(90 年代的主页),现已发展成为动态网页——对话和评论成为常态(博客)。这产生了不断变化的内容，其中讨论的顺序是相关性。谷歌和其他主流搜索引擎目前迎合静态网页的旧时代。虽然这种方式现在运行良好，但未来的搜索和网页导航将比现在的网络更加有用。(显然，Technorati 和其他公司开始向博客提供这种最近搜索服务)。"

**12。[微内容思考:Web 2.0 清单](https://web.archive.org/web/20220817045306/http://www.sivas.com/aleene/microcontent/index.php?id=P2205)**

上周我们错过了这个。定义 web 2.0 服务的不断发展的清单。
 *" *结构化微内容——服务应该能够处理结构化微内容。这可以是存储在服务中或由服务处理的数据；
*外部数据–数据应该主要在外部。因此，服务的主要焦点应该是处理微内容，而不是存储微内容。这也意味着服务应该能够获得外部数据。这可以通过提要(有限窗口)或导入功能来实现。这也意味着用户可以完全控制他的数据(编辑、删除等)。);
*许可证–对于每个微内容项目，用户决定使用许可证。在这里，人们可以区分私有数据(即，仅针对用户的数据，因此具有非常严格的许可)、组数据(其中，针对每个项目确定其他用户可以对其做什么)和公共数据(知识共享许可对其有效)。(备注:我可能会扔掉这个，因为它与外部数据密切相关)；
*提要丰富——一个服务应该有许多客户机可以订阅的提要。这种提要以任何可以想象的方式混合和匹配处理过的微内容。饲料在性质上可以是复合的，并使用许多类型的外壳。这些馈送允许微内容的联合；
* Web API–一个服务应该提供许多 Web API，这些 API 允许将它们的功能集成到其他服务中；
*桌面集成——服务不仅应该存在于网络上，还应该允许与桌面紧密集成。例如，这可以通过微内容客户端来实现。
*单一身份——用户不必在他想使用的所有服务上复制和创建身份。拥有一个可以导出到服务的单一身份(但是多个角色)就足够了；
* micro web–用户应该能够在字段级别上遍历微内容空间。从一个微内容项中的一个字段，用户应该能够转到相关的其他微内容项。用户自己决定什么是相关的。因此，看到一个名字“阿诺·琳”，他应该可以去相关的 FOAF-文件，博客，标签等；
*小写结构——服务应该支持小写的微内容结构。这意味着结构不是预先设定的，而是由用户决定的。没有一成不变的标准制定委员会或服务机构；
